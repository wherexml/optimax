**1. 文档说明与交付原则**

本文件以原始 PRD 中明确的产品定位、行业切入、Agent-first 原则、五层架构和分阶段目标为基础，并将其拆分为更适合研发执行的页面、模块、交互、对象与验收要求。目标不是再写一遍战略，而是把战略意图固化为可以开发、测试、联调和验收的交付说明。

**1.1 产品边界与非目标**

| **维度** | **说明**                                                     |
| -------- | ------------------------------------------------------------ |
| 产品定位 | 叠加在 ERP / SRM / 电子采购 / 计划系统之上的风险 intelligence、业务影响量化与协同处置决策层平台。 |
| 本期范围 | 围绕风险事件主线构建工作台、事件详情、供应商画像、风险地图、War Room、报告中心、规则中心与管理后台。 |
| 非目标   | 不重建 ERP / SRM 主流程，不做通用 BI 平台，不把 Agent 设计成脱离工作流的聊天入口。 |
| 交付原则 | 工作流优先、结构化结果优先、人机协同、权限可控、审计留痕、先打通 P0 再做差异化 P2。 |

 

**1.2 阶段范围（建议）**

| **阶段** | **目标**                   | **主要交付**                                                 | **关键指标**                         |
| -------- | -------------------------- | ------------------------------------------------------------ | ------------------------------------ |
| MVP / P0 | 打通外部风险监测与基础闭环 | 风险工作台、事件详情、基础驾驶舱、基础地图、规则中心、订阅消息、报告初版 | 重大事件首轮预警 ≤ 90 分钟           |
| P1       | 打通组织协同与影响解释     | 供应商画像、War Room、报告中心增强、评分解释、内部规则联动、后台治理 | 闭环率 > 70%，人工筛选时间下降 > 50% |
| P2       | 形成差异化决策能力         | 优化中心、替代 / 调拨模型、全局智能助手增强、方案审批与执行联动 | Top 风险处置时间下降 > 30%           |

 

**1.3 建议纳入里程碑的核心指标**

| **指标域** | **说明**                                  | **目标**  |
| ---------- | ----------------------------------------- | --------- |
| 预警时效   | 重大外部事件到首轮预警                    | ≤ 90 分钟 |
| 映射质量   | Top 100 高价值供应商 / 关键物料关联准确率 | > 85%     |
| 运营效率   | 人工筛选时间下降                          | > 50%     |
| 闭环质量   | 高风险事件闭环率                          | > 70%     |
| 分析效率   | 重大事件完成首轮影响分析                  | ≤ 4 小时  |

 

  **■** **研发落地的五条原则**  **•** 先把风险事件主线做通，再做广。  **•** 所有页面都围绕“发生了什么、影响谁、为什么、下一步做什么”组织。  **•** 所有关键动作都要有权限控制和审计记录。  **•** Agent 输出必须落回结构化结果，不以自然语言作为唯一交付物。  **•** 地图与大屏是结果层，不能牺牲底层实体映射与流程闭环。  

**2. 产品总体架构与产品旅程设计**

产品应围绕一条统一旅程展开：监测与采集、事件归并、实体映射、风险评估、预警推送、协同处置、复盘沉淀。页面层只是不同角色在这条旅程上的入口，研发拆分应围绕同一条对象流和状态流。



图 1 端到端产品旅程示意（从事件发现到复盘沉淀）

**2.3 状态机与人工检查点**

| **对象** | **状态流转**                                                 | **人工检查点**                                           |
| -------- | ------------------------------------------------------------ | -------------------------------------------------------- |
| 风险事件 | 新入库 → 已归并 → 待映射 → 待研判 → 已预警 → 处理中 → 已解决 / 已归档 / 已驳回 | 低置信映射、红色预警升级、重大供应商相关事件需人工确认。 |

| 任务     | 待领取 → 进行中 → 待审批 → 已完成 / 已阻塞 / 已超时          | 涉及成本、交期重大影响的动作必须审批。                   |

 

**2.4 人机协同策略**

**3. 客户使用旅程**

用户旅程需要分别覆盖“外部风险事件”与“内部规则触发”两种主路径，同时再映射到不同角色的高频入口。研发与设计应确保不同角色虽然入口不同，但看到的是同一条事实链与动作链。

**3.1 典型外部风险旅程（关键器件断供 / 港口中断）**

| **阶段**   | **时间**    | **主要角色**                | **关键动作**                     | **系统输出**                            |
| ---------- | ----------- | --------------------------- | -------------------------------- | --------------------------------------- |
| 发现风险   | T0–T+5m     | 采集 Agent / 专家运营员     | 查看新事件、判断是否为重点来源   | 事件卡片、来源可信度、初始标签          |
| 归并与映射 | T+5m–T+30m  | 映射 Agent / 运营员         | 确认供应商、物料、订单、区域关联 | 关联对象、映射置信度、异常提示          |
| 评估与升级 | T+30m–T+90m | 风控经理 / 采购 / 计划      | 判断等级、识别影响范围、是否升级 | 风险等级、影响清单、预警推送、Case 草稿 |
| 协同处置   | T+90m–T+1d  | 采购 / 计划 / 工厂 / 审批人 | 确定替代、调拨、通知和任务分工   | War Room、任务板、审批单、会议纪要      |
| 复盘沉淀   | 事件关闭后  | 运营经理 / 专家运营员       | 评估避免损失、沉淀案例和报告     | 复盘报告、案例标签、模板更新建议        |

 

**3.2 典型内部规则旅程（需求 / 采购 / 执行环节）**

| **环节** | **触发示例**                                   | **用户动作**                         | **平台能力**                 | **结果**               |
| -------- | ---------------------------------------------- | ------------------------------------ | ---------------------------- | ---------------------- |
| 需求风险 | 需求变更频繁、取消率异常、需求与执行偏差       | 确认是否属实、判断是否需要阻断或提醒 | 规则引擎、阈值比对、任务派发 | 形成内部风险事件或待办 |
| 采购风险 | 合同到期、价格异常、评分标准不一致、保证金逾期 | 查看命中规则、发起提醒或审批         | 规则中心、价格联动、流程提醒 | 形成采购风险单         |
| 执行风险 | 订单配置异常、质量预警、超框架金额、超期接收   | 阻断 / 放行、补充原因、追加责任人    | 审批阻断、待办、质检联动     | 进入闭环工单           |
| 闭环处理 | 红黄预警后需要跟催与升级                       | 认领、评论、推进、关闭               | Case、SLA、跟办 Agent、审计  | 形成闭环记录与复盘素材 |

 

**3.3 角色视角旅程矩阵**

| **角色**         | **进入系统的首要目的**                  | **最常用页面**                 | **典型判断点**               | **交付给 TA 的结果**           |
| ---------------- | --------------------------------------- | ------------------------------ | ---------------------------- | ------------------------------ |
| 集团风控平台主管 | 看全局态势与重大事件升级链路            | 首页驾驶舱、风险地图、月报     | 是否升级、是否要求跨区域协同 | 管理摘要、升级视图、闭环率     |
| 供应链运营经理   | 快速确认影响并推动处置                  | 风险工作台、事件详情、War Room | 优先处理谁、先开哪个 Case    | 影响清单、责任链、任务板       |
| 采购经理         | 知道哪个供应商 / 物料有问题以及替代方案 | 供应商画像、事件详情、优化中心 | 是否启动替代、是否需要审批   | 供应商对比、候选方案           |
| 计划经理         | 判断库存、调拨、排产是否受影响          | 事件详情、优化中心、风险地图   | 是否调拨、是否改排产         | 库存影响、调拨建议、受影响订单 |
| 专家运营员       | 提升预警质量并输出周报                  | 风险工作台、报告中心、案例库   | 哪些是噪声、哪些应沉淀模板   | 清洗后的事件池、快报、案例     |

 

  **■** **使用旅程设计原则**  **•** 任何角色都不能只看到“信息”，还必须看到“下一步动作”。  **•** 角色差异主要体现在入口与权限，不应该体现在事实版本不一致。  **•** 外部风险与内部规则是两条不同入口，但最终要落到同一个 Case /  Task / Audit 闭环。

**7. 视觉 UI 设计规范**

本产品是面向国央企与复杂制造客户的运营型平台，视觉风格应强调专业、稳重、信息层级清晰，避免炫技式设计。风格上建议采用“深蓝主色 + 中性背景 + 风险色强调”的体系。

**7.1 设计原则**

**•** 稳重可信：强调业务可信度与审计感，避免过度娱乐化设计。

**•** 操作优先：所有高频页面应减少跳转，优先抽屉、侧栏和就地操作。

**•** 风险可辨：颜色、标签、状态和时间必须清晰区分风险等级与 SLA 状态。

**•** 信息可钻取：管理层看总览，运营层看明细，二者通过相同指标口径联动。

**•** 组件复用：地图卡、指标卡、事件列表、时间线、批量操作条等应形成统一组件库。
